## 1. 研究领域

当前关注方向调整为：

1. `FFGO` 视频定制模型在 `FastGen` 蒸馏框架下的可蒸馏性验证；
2. 基于视频扩散模型的少步加速蒸馏，优先验证 `50 -> 32 step` 的 pipeline；
3. 在保证 teacher/student 接口尽量可对齐的前提下，优先评估 `FastGen` 现成 `Wan2.2 TI2V 5B` 原生框架是否可直接作为 I2V 蒸馏起点，再决定是否进入 `FFGO-compatible teacher path -> student` 的全参数蒸馏改造。

## 2. 领域核心问题

当前要解决的核心问题不是最终画质，而是 **pipeline 是否能跑通**。

具体问题如下：

1. `FFGO` 本身并不是 FastGen 原生支持的 teacher 形态。公开资料显示，FFGO 是建立在 `Wan2.2-I2V-A14B` 上的轻量 LoRA 视频定制方法，主要工作流使用的是其公开仓库中的推理与训练代码，而不是 FastGen 训练体系。

2. 这意味着把 FFGO 接入 FastGen 做蒸馏时，至少存在三类潜在不一致：
   - **模型族不一致**：FFGO 公开基座是 `Wan2.2-I2V-A14B`；
   - **任务类型不一致**：FFGO 是 `I2V` 定制，而我当前服务器上的 FastGen 主实验多围绕 `Wan2.1 T2V`；
   - **adapter 形态不一致**：FFGO 是 `LoRA adapter + base model`，而我当前实际采用的 student 是 `Wan2.1 1.3B 全参数学习`，不考虑 LoRA。

3. 但本周进一步检查服务器代码后发现，`FastGen` 并非完全不支持 `Wan2.2 I2V/TI2V`。它已经原生包含：
   - `WanI2V` 网络封装；
   - `Wan2.2-TI2V-5B-Diffusers` 的 net config；
   - `WanI2V/config_dmd2_wan22_5b.py` 与 `config_sft_wan22_5b.py` 等现成实验配置；
   - I2V 推理脚本入口。

4. 因此当前阶段的核心问题应重新定义为两层：
   - **第一层**：先验证 `FastGen` 现成 `Wan2.2 TI2V 5B` 原生框架在本机环境下能否顺利跑通；
   - **第二层**：在原生 `Wan2.2 TI2V` 路线稳定后，再判断公开 `FFGO (A14B + LoRA)` 需要补哪些接入层；
   - 也就是先回答“FastGen 原生 Wan2.2 I2V 蒸馏能否成立”，再回答“FFGO teacher 能否接进来”。

## 3. 技术方案

### 3.1 FFGO 公开结构调研结论

根据 FFGO 项目主页、GitHub 仓库和公开 LoRA adapter 信息，当前公开版 FFGO 的结构可概括为：

1. **FFGO 是一个基于第一帧定制的 I2V 方法，不是一个重新设计架构的新视频模型。**
   它强调利用视频模型对第一帧的“conceptual memory buffer”能力，用少量样本和轻量 LoRA 适配实现定制化视频生成。

2. **公开基座模型是 `Wan2.2-I2V-A14B`。**
   项目 README 明确要求下载 `Wan2.2-I2V-A14B` 作为基础模型，然后加载 FFGO 的 LoRA adapters 进行推理。

3. **FFGO 公开版采用 LoRA 适配而不是 full finetune。**
   公开 Hugging Face adapter 中可以看到：
   - rank = `128`
   - 训练数据使用约 `50` 个训练视频的子集示例
   - 训练模式为 `i2v`
   - 训练视频长度为 `81` 帧

4. **LoRA 插入位置比较广，不只是 attention qkv。**
   公开 adapter 结构显示它覆盖：
   - `time_projection`
   - self-attention 的 `q/k/v/o`
   - cross-attention 的 `q/k/v/o`
   - FFN 线性层
   这说明 FFGO 的定制能力并不只是 prompt 微调，而是对 Wan2.2 的 3D transformer 主干做了较全面的低秩适配。

5. **公开推理形态包含 low-noise / high-noise LoRA，以及 4-step merged LoRA 推理入口。**
   这意味着 FFGO 自身已经有一部分“少步或更快推理”的公开探索，但它不是 FastGen 路线下的标准蒸馏训练实现。

6. **FastGen 对 `Wan2.2 TI2V 5B` 并非零支持，而是已有原生基础。**
   服务器代码显示：
   - 已有 `WanI2V` 网络实现；
   - 已有 `Wan22_I2V_5B_Config`；
   - 已有 `WanI2V/config_dmd2_wan22_5b.py`；
   - 已有 `WanI2V/config_sft_wan22_5b.py`；
   - 推理脚本也支持 I2V 输入。
   这说明后续不一定要从 `Wan2.1 T2V` placeholder 路线硬扩展，可以先借助现成 `Wan2.2 TI2V 5B` 框架做验证。

### 3.3 预期技术难点

1. `FFGO (Wan2.2-I2V-A14B + LoRA)` 与 FastGen 已有的 `Wan2.2 TI2V 5B` 原生支持并不等价；
2. FastGen 当前看起来原生支持到 `Wan2.2 TI2V 5B` 的 `DMD2 / SFT / I2V inference`，但未必对 `CD / MeanFlow / FFGO LoRA teacher` 都有现成入口；
3. 如果坚持公开 FFGO 路线，则 teacher 仍属于 `Wan2.2 I2V/TI2V` 家族，但其 `A14B + LoRA` 形态和 FastGen 现成 `5B diffusers` config 之间仍有 gap；
4. 如果 FFGO teacher 的定制能力依赖 first-frame 机制和额外 prompt phrase，那么蒸馏时需要确认这些条件是否被完整传递；
5. 因此下周的技术重点不应再是盲目自建 teacher 接口，而应先判断现成 `Wan2.2 TI2V` 框架能复用到什么程度。

## 5. 结论与发现

本周的核心结论如下：

1. **FFGO 是一个具体公开方法，不是泛指“电商 LoRA 视频模型”。**
   它的公开版核心是：基于第一帧组合输入 + LoRA 轻量适配 + Wan2.2-I2V-A14B 基座的视频定制。

2. **FFGO 和当前服务器上的 FastGen 主链并不天然兼容。**
   当前服务器上的 FastGen 主要经验来自 `Wan2.1` 的多种蒸馏方法复现，而 FFGO 的公开实现基于 `Wan2.2-I2V-A14B` 和单独的 LoRA/推理工作流。这意味着当前阶段首先要做的是“构造 FFGO-compatible teacher path”，而不是“直接复用 FFGO 官方训练逻辑”。

3. **但 FastGen 对 `Wan2.2 TI2V 5B` 并不是零支持。**
   本周复查服务器代码后确认，FastGen 已经原生提供：
   - `WanI2V` 网络实现；
   - `Wan22_I2V_5B_Config`；
   - `WanI2V/config_dmd2_wan22_5b.py`；
   - `WanI2V/config_sft_wan22_5b.py`；
   - I2V 推理入口。
   因此更合理的后续路径不是从零自建 `Wan2.2` 支持，而是先基于现成框架验证它在本机环境下的可用性。

4. **当前阶段性成果已经出现：最小闭环已被打通。**
   目前已确认：
   - teacher path 可以通过单独的 `model.teacher.model_id_or_local_path` 接入；
   - `Wan2.1 1.3B` student 可以在 FastGen 下正常完成 2-GPU FSDP 初始化；
   - `32-step` 设定可以进入真实训练循环；
   - `CD 1000 iter` 试运行已刷出首轮 loss，说明 pipeline 不再停留在“只会初始化”的阶段。

5. **当前 student 不用 LoRA 是合理的。**
   因为当前任务是 step distillation feasibility，而不是 adapter distillation。student 采用 `Wan2.1 1.3B` 全参数训练，更有利于把问题收敛到“少步蒸馏能否成立”，而不是 LoRA 容量是否足够。

     - `config_dmd2_wan22_5b.py`；
     - I2V 推理入口。
   - 因此后续应先回答：现成原生 `Wan2.2 TI2V` 路线在本机环境下能否直接作为蒸馏起点。

2. **中期目标：如果原生 `Wan2.2 TI2V` 路线可用，再评估如何接真实 FFGO teacher。**
   - 目标不是立刻追最终业务效果，而是先把：
     - `真实 FFGO teacher`
     - `FastGen 训练侧`
     - `Wan student`
     三者接成一条可重复运行的训练链。

3. **如果内部 FFGO 可以整理成 `Wan2.1-compatible merged teacher`。**
   - 这是最低风险方案。
   - 做法是直接替换当前 `FFGO_TEACHER_PATH`，复用现有：
     - `DMD2 / CD` 配置
     - `32-step student`
     - 当前已经跑通的训练与推理验证流程
   - 这样可以最快回答：真实 FFGO teacher 的能力能否在当前 FastGen 框架下被蒸到 student 中。

4. **公开路线：如果 FFGO 保持 `Wan2.2-I2V-A14B + LoRA` 形态。**
   - 这时不能只替换权重路径，而要补完整的 teacher 接入层：
     - 增加 `LoRA adapter / merged weight` 加载能力；
     - 增加 `Wan2.2 I2V/TI2V` 兼容 config；
     - 把 dataloader 从当前的 `T2V(mp4 + txt)` 扩展到 FFGO 所需条件；
     - 对齐 teacher forward 所需的 `first frame / ref image / prompt phrase` 等输入。
   - 如果后续走向编辑任务，则还要继续扩展：
     - `source video`
     - `mask`
     - identity / background / temporal 相关约束

5. **student 侧的预期。**
   - 当前 student 仍以 `Wan2.1 1.3B` 全参数训练为主，用来验证蒸馏 feasibility。
   - 如果 teacher 最终确认属于 `Wan2.2 I2V` 家族，则后续需要在两者之间做选择：
     - 要么把 student 切到兼容 backbone；
     - 要么补跨架构蒸馏对齐逻辑。
   - 当前不建议同时引入 `LoRA student`，避免把问题从“step distillation”带偏到“adapter capacity”。

6. **整合完成后的阶段目标。**
   - Stage 0: 现成 `Wan2.2 TI2V 5B` 原生 config 在本机环境下跑通；
   - Stage A: 判断现成 `WanI2V` 路线能否改到更接近 `50 -> 32` 的实验设定；
   - Stage B: 再决定是否接入真实 FFGO teacher，并确认其单独推理可用；
   - Stage C: teacher 可以被 FastGen teacher side 正常调用；
   - Stage D: `50 -> 32` smoke run 跑通；
   - Stage E: `5000 iter / 500 ckpt` 正式 run 跑通；
   - Stage F: 再决定是否继续做 `32 -> 16 -> 8 -> 4` progressive distillation。

7. **最终希望得到的产物。**
   - 一套可复现的 FFGO teacher 接入方式；
   - 一条稳定的 FastGen 蒸馏训练脚本；
   - 一组固定 prompt / 固定条件下的 checkpoint 对比资产；
   - 一个明确结论：真实 FFGO 更适合 `DMD2`、`CD`，还是需要新的方法。

### 当前判断

如果当前阶段目标是 **先打通 pipeline，再进入更正式训练**，那么我当前的判断是：

1. 先不把“数据质量、电商效果、最终业务指标”放进目标函数；
2. 先验证现成 `Wan2.2 TI2V 5B` 原生框架在本机环境下是否可直接复用；
3. 再验证 FFGO-compatible teacher path 能否被 FastGen teacher side 正常消费；
4. 已经先做 `50 -> 32 step` 的 `CD 1000 iter` smoke run；
5. 当前已经切到 `DMD2 5000 iter / 500 ckpt`，把实验从“能跑”推进到“开始形成阶段结果”。

### 未来整合判断

如果后续要把 FFGO 真正整合进当前框架，我更倾向于分两种情况处理：

1. **先跑现成 `Wan2.2 TI2V 5B` 原生路线。**
   - 这是当前发现后最应该优先测试的方向；
   - 它可以帮助判断：FastGen 原生 I2V 蒸馏到底能直接用到什么程度。

2. **在原生 `Wan2.2 TI2V` 路线可用后，再评估 FFGO teacher 接入。**
   - 这样可以把问题拆成两层：
     - FastGen 原生 I2V 蒸馏能不能跑；
     - FFGO teacher 形态需不需要额外适配。

3. **内部 FFGO 能整理成 `Wan2.1-compatible merged teacher`。**
   - 这是最优路径；
   - 可以直接沿用当前已经稳定的：
     - `teacher path`
     - `DMD2 / CD`
     - `32-step student`
   - 这样能最快把问题收敛到“FFGO teacher 的能力能否被蒸馏”。

4. **FFGO 只能保持公开 `Wan2.2-I2V + LoRA` 路线。**
   - 那就不能把它当成简单换权重；
   - 需要新增：
     - teacher adapter/merge 加载逻辑；
     - `Wan2.2` 系列 config；
     - `I2V / first-frame customization` 数据与条件处理；
     - 更长的 teacher-student I/O 对齐验证。

因此当前前置实验的意义是：先把 FFGO 之外的蒸馏基础设施做成稳定底座，后面无论走哪条 FFGO 接入路线，都不是从零开始。
